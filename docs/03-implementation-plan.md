# 任务管理工具实施计划

## 1. 文档目的

本文档用于将需求规格和详细设计拆分为可执行的开发任务，明确阶段目标、依赖关系、交付物和验收条件，作为后续逐项实施的执行清单。

关联文档：

- [`01-requirements-spec.md`](R:\Files\Workspace\Mine\Todo\docs\01-requirements-spec.md)
- [`02-detailed-design.md`](R:\Files\Workspace\Mine\Todo\docs\02-detailed-design.md)

## 2. 实施原则

- 优先建立稳定的数据模型和规则引擎，再推进复杂界面。
- 优先完成本地离线可用能力，再接入同步。
- 先做核心闭环，再补后台、提醒和平台增强能力。
- 每个阶段都应形成可运行、可验证的阶段性结果。
- 所有任务拆分尽量保持单一职责，避免跨模块大杂烩式开发。

## 3. 总体阶段划分

建议按以下阶段推进：

1. 项目初始化与基础框架
2. 本地数据模型与存储
3. 任务基础能力
4. 重复规则与实例展开
5. 单次覆盖与模板版本段
6. 日历视图与近期视图
7. 危险日与工作日计算
8. 标签、设置与辅助页面
9. WebDAV 同步
10. 后台运行与提醒
11. 收尾与质量提升

## 4. 阶段计划

### 阶段 1. 项目初始化与基础框架

目标：

- 建立 Flutter 项目基础结构
- 明确目录规范、状态管理方案、数据库接入方式
- 搭建基础页面骨架和导航结构

任务：

- 创建 Flutter 工程
- 建立目录结构
- 选定状态管理方案
- 接入 SQLite 访问方案
- 接入本地时间处理库
- 接入日志与错误捕获基础能力
- 搭建应用壳、路由和基础主题

- 可运行的 Flutter 初始工程
- 基础目录结构

- 数据访问基础封装

- 应用具备基础页面切换能力
- 数据库初始化流程可运行

- 实现数据库迁移机制
- 实现任务模板仓储
- 实现模板版本段仓储
- 实现单次覆盖仓储
- 实现标签仓储
- 实现节假日仓储
- 实现同步元数据仓储

交付物：

- 数据库建表脚本或初始化代码
- 各核心实体仓储层
- 基础 CRUD 单元测试

验收条件：

- 本地数据库可初始化
- 所有核心实体可完成增删改查
- 数据迁移具备可扩展能力

### 阶段 3. 任务基础能力

目标：

- 完成非重复任务的创建、编辑、删除、状态修改、详情查看

- 任务详情页首版
- 基础任务用例服务

验收条件：

- 用户可完整创建和编辑非重复任务
- 用户可修改任务状态
- 任务数据在重启后仍能正确显示

### 阶段 4. 重复规则与实例展开

## 5. 推荐任务粒度

后续每个开发任务建议控制在以下粒度：

- 单个任务只聚焦一个模块或一条核心能力
- 单个任务最好能在一次开发周期内完成并验证
- 复杂能力拆为“数据层任务 + 领域层任务 + 界面任务”

示例拆分方式：

- 重复规则日重复展开
- 重复规则周重复展开
- 单次覆盖合并逻辑
- 日历单日分组渲染
- 近期任务排序服务

## 6. 依赖关系

主要依赖链如下：

1. 项目初始化 -> 数据模型与存储
2. 数据模型与存储 -> 任务基础能力
3. 任务基础能力 -> 重复规则与实例展开
4. 重复规则与实例展开 -> 单次覆盖与模板版本段
5. 重复规则与单次覆盖 -> 日历视图与近期视图
6. 节假日与危险日计算 -> 近期排序与提醒
7. 本地完整能力稳定后 -> WebDAV 同步
8. 同步稳定后 -> 后台与提醒增强

说明：

- 同步不应过早介入，否则会放大本地模型未稳定带来的返工成本。
- 日历视图应在实例展开和覆盖合并稳定后再重点开发。

## 8. 测试策略

建议测试按以下层次推进：

- 单元测试
  覆盖重复规则、危险日计算、工作日倒推、覆盖合并、排序逻辑。
- 仓储测试
  覆盖数据库读写、迁移、数据恢复。
- 集成测试
  覆盖任务创建到展示的完整链路、同步前检查、模板修改与覆盖保留。
- 手工验证
  覆盖复杂时间场景、节假日样例、界面表现和多端同步流程。

重点测试场景：

- 每月固定日期重复 + 节假日工作日倒推
- 单次实例延期任务日和危险日
- 模板从指定日期开始修改
- 已有未来覆盖时再次修改模板
- 跳过本次
- 已完成和已取消任务的展示顺序
- 近期视图未来 31 天排序
- 保存前远端发生变化

## 9. 风险与控制

### 9.1 规则引擎复杂度风险

风险：

- 重复规则、版本段和覆盖叠加后容易出现边界错误。

控制：

- 先完成纯领域层测试，再接界面。

### 9.2 同步返工风险

风险：

- 本地模型不稳定时过早接入同步，会导致数据结构频繁变更。

控制：

- 将同步放在本地完整能力基本稳定后接入。

### 9.3 日历性能风险

风险：

- 日期流很长，且单日可能有很多任务。

控制：

- 强制使用按窗口查询和虚拟化渲染。

### 9.4 平台能力差异风险

风险：

- Windows 后台与 Android 后台行为不同。

控制：

- 保持领域逻辑纯 Dart，平台层只做能力适配。

1. 初始化 Flutter 工程和目录结构
