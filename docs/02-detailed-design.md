# 任务管理工具详细设计

当前状态：`已完成`

## 1. 文档目的

本文档基于 [`01-requirements-spec.md`](R:\Files\Workspace\Mine\Todo\docs\01-requirements-spec.md) 编写，用于明确当前主线的技术架构、核心模块、数据模型、同步方案、后台策略与双端兼容策略，作为后续实施计划与编码实现的直接依据。

本文档聚焦“如何实现”，并以当前已确认的业务前提为基础：

- 任务以截止时间为核心
- 开始时间表示进入视野时间，不表示持续占用
- 同步采用远程优先语义
- Windows 与 Android 均为正式目标平台

## 2. 设计原则

- 业务语义优先于平台交互形式，双端允许不同交互，但不得改变任务含义。
- 核心业务、数据模型、同步逻辑统一收敛在共享核心层，不在前端页面中分散实现。
- 本地优先仍然成立，但远程优先决定冲突处理结果。
- 同步策略优先简单、可恢复、可解释，不追求复杂协同算法。
- Windows 与 Android 从设计阶段就按双端兼容建模，不接受单端写死后再补适配。
- 后台低资源占用通过架构约束实现，而不是依赖前端窗口长期存活。
- 前端整体界面以自定义设计系统为主，不引入大型 UI 库作为默认基座。

## 3. 技术路线

### 3.1 总体选型

当前主线采用以下技术架构：

- 应用壳：`Tauri v2`
- 共享核心：`Rust`
- 前端：`Vue 3 + TypeScript + Vite`
- 本地存储：`SQLite`
- 同步协议：`WebDAV`
- 数据序列化：`serde`
- 日志：`tracing`
- 异步运行时：`tokio`

选型理由如下：

- `Tauri` 能同时覆盖 Windows 与 Android，并允许将数据库、WebDAV、后台能力放在 Rust 侧统一实现。
- `Rust` 适合作为共享核心，便于统一实现任务规则、同步逻辑和平台能力抽象。
- 前端采用 `Vue 3` 与 Web 技术栈，便于分别优化桌面与移动端交互，并更贴合中文开发环境下的使用习惯。
- `SQLite` 适合作为本地结构化数据存储。
- `WebDAV` 继续保留，但只由 Rust 同步层访问，前端不直接连接 WebDAV。

### 3.2 前端约束

前端层采用以下约束：

- 使用 `Vue 3 + TypeScript` 组织页面与交互
- 使用 `Vite` 作为前端构建工具
- 使用 `CSS Variables + CSS Modules` 或同等粒度样式方案构建设计系统
- 不引入大型跨平台 UI 组件库作为默认基座
- 查询型数据状态优先使用 `@tanstack/vue-query`
- 短期页面状态与交互状态优先使用 `Pinia`
- 复杂日期时间控件允许例外引入 `Naive UI` 的 `DatePicker` / `DateTimePicker`

约束说明：

- 任务、同步、提醒、数据库访问不允许直接写在前端页面中。
- 前端只通过 Tauri command 与核心层交互。
- 同一业务语义在桌面与移动端可以使用不同控件，但字段结构、保存流程和结果解释必须一致。
- `Naive UI` 仅作为复杂表单控件补充来源，不作为整站 UI 基座。

### 3.3 Rust 核心约束

Rust 核心层负责以下能力：

- SQLite 访问与迁移
- 任务规则与领域服务
- 重复实例展开
- 单次覆盖合并
- 危险日与工作日计算
- WebDAV 同步
- 后台调度
- 平台能力适配

约束说明：

- 前端不得直接操作 SQLite 文件。
- 同步只允许由 Rust 侧执行。
- 与平台相关的能力统一封装在平台适配层，不允许业务层直接依赖平台 API。

## 4. 总体架构

### 4.1 分层结构

系统按以下层次组织：

- 表现层
  Vue 页面、组件、布局、输入校验、交互反馈。
- 应用层
  Tauri command、用例编排、事务边界、错误映射。
- 领域层
  任务规则、重复展开、覆盖合并、危险日计算、排序投影。
- 基础设施层
  SQLite、WebDAV、日志、文件系统、配置、通知。
- 平台适配层
  Windows 托盘、窗口生命周期、Android 生命周期、权限与通知桥接。

### 4.2 仓库结构建议

建议采用如下结构：

前端：

- `src/app`
- `src/pages`
- `src/features`
- `src/entities`
- `src/shared`

Rust：

- `src-tauri/src/app`
- `src-tauri/src/domain`
- `src-tauri/src/infra`
- `src-tauri/src/platform`
- `src-tauri/src/commands`

### 4.3 核心模块

建议划分为以下模块：

- `task_core`
  任务、状态、优先级、标签等基础模型。
- `recurrence_engine`
  重复规则、实例展开、稳定实例键生成。
- `override_engine`
  单次覆盖、实例状态覆盖、模板覆盖合并。
- `projection_service`
  日历视图、近期视图、详情视图投影。
- `danger_service`
  危险日计算、工作日倒推。
- `holiday_service`
  节假日与调休数据管理。
- `sync_service`
  WebDAV 元数据检查、远端拉取、本地上传、冲突处理。
- `reminder_service`
  本地提醒计划生成与更新。
- `platform_service`
  托盘、窗口、前后台生命周期、平台通知。
- `settings_service`
  同步配置、显示偏好、后台策略等设置项。

## 5. 数据模型设计

本系统继续采用“模板 + 模板版本段 + 单次覆盖”的数据模型，不采用未来实例全量落库。

### 5.1 核心表

建议保留以下核心表：

- `task_series`
- `task_series_revision`
- `task_occurrence_override`
- `tag`
- `holiday_calendar`
- `sync_meta`
- `app_settings`

必要时可增加：

- `reminder_state`
- `local_change_journal`
- `sync_recovery_record`

### 5.2 任务模板 `task_series`

用于描述任务的稳定主身份。

建议字段：

- `id`
- `kind`
- `created_at`
- `updated_at`
- `archived_at`

说明：

- `kind` 用于区分单次任务与重复任务。
- 与展示直接相关的业务字段主要放在版本段中，避免未来模板修改污染历史。

### 5.3 模板版本段 `task_series_revision`

用于承载某个时间区间内生效的任务定义。

建议字段：

- `id`
- `series_id`
- `effective_from`
- `effective_until`
- `title`
- `note`
- `tag_id`
- `priority`
- `all_day`
- `start_at_time_part`
- `due_at_time_part`
- `duration_seconds`
- `recurrence_type`
- `recurrence_interval`
- `recurrence_rule_json`
- `recurrence_until`
- `danger_offset_value`
- `danger_offset_unit`
- `danger_use_workday`
- `created_at`
- `updated_at`

说明：

- 同一个任务模板允许存在多个版本段。
- 版本段只影响其生效时间之后的实例。
- 单次任务可视为只有一个版本段且只展开一次。

### 5.4 单次覆盖 `task_occurrence_override`

用于记录某个具体实例的独立修改。

建议字段：

- `id`
- `series_id`
- `occurrence_key`
- `override_start_at`
- `override_due_at`
- `override_danger_at`
- `override_title`
- `override_note`
- `override_tag_id`
- `override_priority`
- `status`
- `completed_at`
- `cancelled_at`
- `detached_as_single`
- `created_at`
- `updated_at`

说明：

- “跳过本次”直接写为 `status = cancelled`。
- 已覆盖实例仍可继续编辑，但落库仍合并为该实例的一条覆盖记录。

### 5.5 标签 `tag`

建议字段：

- `id`
- `name`
- `color_value`
- `sort_order`
- `created_at`
- `updated_at`

### 5.6 节假日 `holiday_calendar`

建议字段：

- `date`
- `day_type`
- `name`
- `source`
- `updated_at`

### 5.7 同步元数据 `sync_meta`

用于记录同步状态与远端基线。

建议字段：

- `key`
- `value`
- `updated_at`

至少保存以下信息：

- `remote_etag`
- `remote_version`
- `last_sync_at`
- `local_dirty`
- `dirty_base_remote_etag`
- `last_sync_result`
- `device_id`

### 5.8 本地变更日志 `local_change_journal`

该表为可选但推荐实现，用于提升离线写入与恢复能力。

建议字段：

- `id`
- `change_type`
- `entity_type`
- `entity_id`
- `payload_json`
- `created_at`

说明：

- 在线且远端未变化时，可在成功写入后清空或压缩。
- 离线写入或远端冲突恢复时，可作为恢复和审计依据。

## 6. 领域模型与规则

### 6.1 核心领域对象

建议定义以下领域对象：

- `TaskSeries`
- `TaskSeriesRevision`
- `TaskOccurrence`
- `TaskOccurrenceOverride`
- `DangerRule`
- `RecurrenceRule`
- `Tag`
- `HolidayInfo`
- `SyncState`

### 6.2 `TaskOccurrence` 投影

`TaskOccurrence` 不要求落库，而是在查询时展开得到。

建议字段：

- `series_id`
- `revision_id`
- `occurrence_key`
- `title`
- `note`
- `tag`
- `priority`
- `status`
- `start_at`
- `due_at`
- `danger_at`
- `all_day`
- `is_override`

### 6.3 时间语义

统一规则如下：

- `due_at` 是排序、提醒、完成判断的核心锚点。
- `start_at` 决定任务何时进入用户视野。
- 跨天任务不表示该任务持续占满整个区间。
- 日历展示可以体现开始与截止，但业务判断仍以截止时间为核心。

### 6.4 重复规则与实例键

重复类型支持：

- 小时
- 天
- 周
- 月
- 年

实例展开流程：

1. 读取与查询窗口有关的版本段。
2. 按规则仅展开查询窗口内相关实例。
3. 为实例生成稳定 `occurrence_key`。
4. 读取覆盖记录。
5. 合并模板实例与覆盖记录。

`occurrence_key` 建议由以下信息稳定编码：

- `series_id`
- `revision_id`
- 原始计划开始时间
- 原始计划截止时间

### 6.5 模板修改规则

当用户选择“修改模板并影响未来”时：

1. 选定生效日期。
2. 截断当前生效中的版本段。
3. 创建新的版本段。
4. 历史实例继续按旧版本解释。
5. 未来已存在覆盖默认保留。

### 6.6 危险日规则

危险日以截止时间为锚点倒推。

支持：

- 提前若干小时
- 提前若干天
- 按自然日倒推
- 按工作日倒推

若实例存在单次危险日覆盖，则该实例不再按模板重新计算危险日。

## 7. 查询与投影设计

### 7.1 日历视图投影

日历视图采用按窗口增量查询：

1. 计算当前可视区日期范围。
2. 向前后预取一定缓冲区间。
3. 展开该区间内实例。
4. 按天聚合投影。
5. 返回前端渲染。

要求：

- 不全量生成未来所有实例。
- 同日任务尽量完整展示。
- 任务状态、危险日、标签颜色在投影阶段就准备好。

### 7.2 近期视图投影

近期视图固定查询：

- 当前时刻起未来 31 天

建议排序键：

1. 状态分组
2. 优先级
3. 危险日临近程度
4. 截止时间
5. 开始时间
6. 创建时间

### 7.3 详情与编辑投影

编辑态与详情态使用单独投影对象，不直接暴露原始数据库结构给前端。

要求：

- 前端拿到的是字段语义完整的 DTO
- 前端不自行拼接模板、版本段和覆盖关系

## 8. 前端界面设计

### 8.1 总体策略

前端界面采用“统一语义，分端交互”的设计。

统一内容：

- 页面信息架构
- 字段语义
- 保存逻辑
- 状态解释

允许差异：

- 导航形态
- 表单控件形态
- 弹层与详情呈现
- 快捷键与手势

### 8.2 页面结构

首版页面包括：

- 日历页
- 近期页
- 任务详情页
- 新建/编辑任务页
- 标签管理页
- 设置页
- 同步状态页

最终信息架构约束：

- `日历页` 与 `近期页` 是两个主视图，不以“编辑表单同屏常驻”作为默认成品形态。
- `任务详情页` 与 `新建/编辑任务页` 属于任务操作流，不等同于主导航中的长期常驻双栏内容区。
- Windows 端最终应围绕“导航 + 主视图 + 详情/编辑面板或独立页”组织，而不是默认固化为当前阶段性“近期入口 + 编辑表单”双栏页。
- Android 端最终应围绕“主视图列表 + 详情/编辑的层级跳转或全屏页”组织，不默认沿用桌面双栏布局。
- 当前阶段 3 的 `AppShell` 仅用于验证任务基础能力闭环、表单字段语义与跨端交互，不作为最终产品页面结构基线。

### 8.3 Windows 交互原则

Windows 端优先：

- 左侧导航
- 高信息密度布局
- 主视图与详情/编辑关系清晰，可采用同屏分栏或独立面板
- 日期使用弹出式日历面板选择
- 时间使用紧凑输入或步进调节
- 键盘快捷键
- 右键与悬停辅助操作
- 主窗口关闭后转后台

### 8.4 Android 交互原则

Android 端优先：

- 底部导航或移动端层级导航
- 主视图、详情、编辑之间采用更明确的页面层级，不默认要求同屏并列
- 更大的触控点击区
- 日期时间使用触屏友好的对话框或底部弹层选择器
- 全屏表单与弹层
- 不依赖 hover 与右键

### 8.5 设计系统约束

界面层要求如下：

- 使用统一的设计 token，例如颜色、字号、间距、圆角、阴影、层级。
- 桌面与移动端可以有不同 token 取值，但命名保持一致。
- 不依赖第三方大型组件库提供默认视觉。
- 组件优先做成语义组件，而不是按具体平台样式硬编码。
- 日期时间组件优先封装统一业务接口，内部复用 `Naive UI` 日期控件能力并分别适配双端展示方式。

## 9. 同步设计

### 9.1 基本原则

同步采用远程优先策略。

含义如下：

- 远端是冲突时的最终基准。
- 本地是工作副本与离线缓存。
- 本地修改与远端冲突时，优先保留远端。

### 9.2 远端文件组织

建议远端维护两类固定文件：

- `todo.meta.json`
- `todo.data.sqlite3`

其中：

- `todo.meta.json` 保存版本号、更新时间、校验值、设备标识、schema 版本。
- `todo.data.sqlite3` 保存当前完整数据快照。

选择整库快照而不是拆分大量小文件，原因如下：

- 简化同步逻辑
- 减少文件级冲突
- 降低远端目录管理复杂度
- 更符合远程优先覆盖策略

### 9.3 同步触发时机

同步触发时机包括：

- 应用启动时
- 应用进入前台时
- 用户手动触发时
- 用户保存前在线检查时
- Windows 后台定时同步时

### 9.4 在线保存流程

在线保存流程如下：

1. 用户在前端发起保存。
2. Rust 应用层先检查远端 `todo.meta.json`。
3. 若远端版本与本地同步基线一致，则允许本地事务提交。
4. 提交后生成新的本地快照与元数据。
5. 上传新的 `todo.data.sqlite3`。
6. 上传新的 `todo.meta.json`。
7. 更新本地 `sync_meta`。

### 9.5 离线保存流程

离线场景下允许本地保存，但必须标记为“待远端确认”。

流程如下：

1. 本地直接提交写入。
2. 记录 `local_dirty = true`。
3. 保存 `dirty_base_remote_etag`。
4. 在下次联网同步时先检查远端。

若下次联网时发现远端未变化：

- 上传本地快照
- 清除 dirty 标记

若下次联网时发现远端已变化：

- 将当前本地脏数据导出为恢复副本
- 下载远端快照覆盖本地
- 记录冲突恢复日志
- 提示用户本地离线改动已被远端覆盖，可从恢复副本查看

### 9.6 启动与恢复同步流程

应用启动或回到前台时：

1. 检查远端元数据。
2. 若远端版本新于本地同步基线，则下载远端快照。
3. 原子替换本地数据库副本。
4. 重新加载投影缓存。
5. 重新计算提醒计划。

### 9.7 同步失败处理

同步失败时：

- 不回滚已成功落地的本地数据库事务。
- 保留本地 dirty 标记。
- 在设置页和同步状态页展示最近一次失败原因。
- Windows 后台与前端均可触发后续重试。

## 10. 后台与提醒设计

### 10.1 后台目标

后台设计必须满足以下目标：

- Windows 长期后台驻留时低资源占用
- 不依赖前端窗口长期存活
- 提醒与同步尽量由 Rust 后台独立执行

### 10.2 Windows 后台模型

Windows 端采用以下模型：

- 主窗口关闭时默认销毁窗口，而不是仅隐藏窗口
- Rust 核心进程与托盘保留
- 托盘用于唤起窗口、查看同步状态、手动同步、退出应用
- 需要展示界面时重新创建主窗口

这样做的目的：

- 不长期保留 WebView 前端页面
- 降低后台常驻内存
- 保留同步、提醒和托盘能力

### 10.3 Android 后台模型

Android 端不假设长期后台常驻。

策略如下：

- 前台使用时完成主要同步和数据刷新
- 本地提醒依赖已计算好的提醒计划
- 后台同步仅作为可选增强，不作为核心一致性前提
- 不依赖类似桌面托盘的交互方式

### 10.4 提醒流程

提醒流程如下：

1. 本地数据变更或同步完成后，重新计算近期危险日与截止提醒。
2. 写入提醒计划状态。
3. 由平台适配层注册或更新本地通知。
4. 通知触发后可唤起应用到指定任务详情。

## 11. 平台兼容策略

### 11.1 共享与差异边界

共享部分：

- 领域模型
- SQLite 结构
- 同步逻辑
- 危险日与重复规则
- DTO 与 command 接口

平台差异部分：

- 导航样式
- 日期时间输入控件
- 托盘与窗口管理
- 通知权限与系统桥接
- 生命周期事件来源

### 11.2 平台能力抽象

平台能力统一通过 `platform_service` 暴露以下接口：

- `show_main_window`
- `close_main_window`
- `schedule_notifications`
- `cancel_notifications`
- `get_lifecycle_state`
- `request_notification_permission`
- `run_background_sync_if_allowed`

要求：

- 业务层不关心当前运行在 Windows 还是 Android。
- 不支持的平台能力必须有显式降级行为，而不是静默失败。

## 12. Tauri Command 设计

前端与 Rust 之间建议通过命令接口通信，至少包括以下类别：

任务：

- `task_create`
- `task_update`
- `task_delete`
- `task_set_status`
- `task_get_detail`

查询：

- `calendar_query`
- `upcoming_query`
- `tag_list`
- `settings_get`

同步：

- `sync_status_get`
- `sync_run`
- `sync_check_before_save`

系统：

- `window_show_main`
- `app_get_capabilities`
- `notification_permission_request`

要求：

- command 返回结构化 DTO，不直接暴露数据库行结构。
- 错误统一映射为前端可展示的业务错误码与错误消息。

## 13. 安全与可靠性

- WebDAV 凭证只保存在 Rust 侧受控配置中，前端不直接处理原始认证流程。
- 本地数据库替换必须采用临时文件 + 原子切换，避免损坏当前可用库。
- 每次远端覆盖本地前，应保留最近一次本地恢复副本。
- schema 版本变化必须通过迁移机制显式处理。

## 14. 性能设计

- 重复实例只按查询窗口展开。
- 前端列表采用虚拟化或分段渲染。
- 后端对近期投影、节假日计算和提醒计划做短期缓存。
- Windows 后台默认不保活前端窗口。
- 同步采用整库快照，避免高频小文件请求。

## 15. 错误处理与边界

### 15.1 时间边界

- 全日任务与定时任务使用统一内部时间模型。
- 需要显式处理时区和夏令时变化。
- 展示层与存储层时间格式分离。

### 15.2 同步边界

- 远端元数据损坏时不得直接覆盖本地。
- 远端快照校验失败时应保留本地并提示重试。
- 离线脏数据在被远端覆盖前必须先生成恢复副本。

### 15.3 平台边界

- 不把托盘、窗口大小、右键菜单等桌面能力写入核心业务流程。
- Android 不支持的后台行为必须有清晰降级路径。

- Rust 单元测试

- Rust 集成测试

- 前端组件测试

- 模板从指定日期开始修改

- 危险日按工作日倒推
- Windows 关闭窗口后后台同步与提醒
- Android 前后台切换后的同步恢复
