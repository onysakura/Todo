# 任务管理工具详细设计

## 1. 文档目的

本文档用于定义任务管理工具的实现方案，明确系统架构、核心模块、数据模型、关键业务流程、同步策略与界面支撑逻辑，为后续实施计划和编码开发提供依据。

本文档基于 [`01-requirements-spec.md`](R:\Files\Workspace\Mine\Todo\docs\01-requirements-spec.md) 编写，聚焦“如何实现”。

## 2. 设计原则

- 以截止时间为核心建模，避免引入不必要的持续事件语义。
- 以本地优先为基础，所有核心功能在离线状态下也应可用。
- 通过模板展开而非全量预生成，降低存储和计算压力。
- 通过“模板版本段 + 单次覆盖”处理未来变更。
- 规则引擎与界面渲染解耦，确保跨平台一致性。
- 同步逻辑尽量简单，优先保证一致性而非复杂冲突合并。

## 3. 总体架构

### 3.1 技术路线

- 客户端框架：Flutter
- 路由：go_router
- 本地存储：SQLite
- 状态管理：flutter_riverpod
- 日志：logging
- 时间处理基础库：intl + timezone
- 数据库访问层：drift
- SQLite 运行时支持：sqlite3_flutter_libs
- 核心业务逻辑：纯 Dart 模块
- 同步方式：WebDAV
- 目标平台：Windows、Android

### 3.2 分层结构

系统按以下层次组织：

- 表现层
  负责页面、组件、交互、状态展示和用户输入。
- 应用层
  负责用例编排，例如创建任务、修改模板、标记完成、执行同步。
- 领域层
  负责任务规则、重复展开、危险日计算、排序与实例覆盖合并。
- 数据层
  负责 SQLite 持久化、本地缓存、同步读写、节假日数据访问。
- 平台适配层
  负责通知、后台常驻、托盘、开机启动等平台能力。

### 3.3 模块划分

建议划分为以下核心模块：

- `task_core`
  任务实体、状态、优先级、标签等基础模型。
- `recurrence_engine`
  重复规则定义、实例展开、模板版本段生效逻辑。
- `override_engine`
  单次覆盖、实例状态覆盖、危险日调整。
- `calendar_projection`
  针对日历视图和近期视图进行区间查询、排序和展示投影。
- `holiday_service`
  节假日和工作日判断。
- `sync_service`
  WebDAV 数据拉取、远端覆盖本地、保存前检查。
- `notification_service`
  本地提醒与后台任务协同。
- `settings_service`
  同步配置、显示偏好、后台选项等。

### 3.4 当前已确定的工程约定

- Dart 包名使用 `todo_app`
- 当前阶段先建立应用壳与目录骨架，再进入数据层实现
- 项目状态管理统一基于 Riverpod
- 项目页面路由统一基于 go_router
- 项目日志基础设施统一基于 logging
- 项目时间处理基础设施统一基于 intl + timezone
- 项目本地数据库统一基于 Drift + SQLite
- 当前构建环境下，`sqlite3` 优先使用系统 SQLite 库以规避构建阶段的网络下载风险

## 4. 核心数据模型

本系统不采用“未来实例全量落库”方案，而采用“模板存储 + 查询时展开 + 覆盖修正”的模型。

### 4.1 任务模板 `task_series`

用于描述一组任务实例的基础定义。单次任务也可视为仅生成一次的模板。

建议字段如下：

- `id`
- `title`
- `note`
- `tag_id`
- `priority`
- `all_day`
- `start_at`
- `due_at`
- `status_default`
- `danger_offset_value`
- `danger_offset_unit`
- `danger_use_workday`
- `created_at`
- `updated_at`
- `archived_at`

设计说明：

- `start_at` 表示进入视野时间。
- `due_at` 表示截止时间，是核心排序和提醒锚点。
- `status_default` 默认取未完成。
- `archived_at` 预留，不作为当前业务重点。

### 4.2 模板版本段 `task_series_revision`

用于支持“从某个生效日期开始修改模板”的需求。

建议字段如下：

- `id`
- `series_id`
- `effective_from`
- `effective_until`
- `title`
- `note`
- `tag_id`
- `priority`
- `all_day`
- `start_at_time_part`
- `due_at_time_part`
- `duration_seconds`
- `recurrence_type`
- `recurrence_interval`
- `recurrence_rule_json`
- `recurrence_until`
- `danger_offset_value`
- `danger_offset_unit`
- `danger_use_workday`
- `created_at`
- `updated_at`

设计说明：

- 同一个任务模板可以存在多个版本段。
- 每次修改模板并影响未来任务时，不直接覆盖历史，而是新增版本段。
- `effective_from` 是模板生效起点。
- `effective_until` 用于关闭旧版本段。
- 时间相关字段拆分为“时间部分 + 时长”有助于重复任务在不同日期上展开。

### 4.3 单次覆盖 `task_occurrence_override`

用于记录某一次实例的特殊变更。

建议字段如下：

- `id`
- `series_id`
- `occurrence_key`
- `override_type`
- `override_start_at`
- `override_due_at`
- `override_danger_at`
- `override_title`
- `override_note`
- `override_tag_id`
- `override_priority`
- `status`
- `detached_as_single`
- `created_at`
- `updated_at`

设计说明：

- `occurrence_key` 用于唯一标识某次模板实例。
- `override_type` 可表示普通修改、取消、独立分离等。
- 若该实例只被取消，则主要记录状态即可。
- 若该实例需脱离模板继续存在，可通过 `detached_as_single` 或后续独立模板化处理。

### 4.4 任务完成记录

由于任务状态是实例级别，应支持实例完成时间记录。可并入覆盖表，也可拆分为独立状态表。

建议优先采用并入覆盖表的方式，避免重复维护实例主键。

必要字段：

- `status`
- `completed_at`
- `cancelled_at`

### 4.5 标签 `tag`

建议字段如下：

- `id`
- `name`
- `color_value`
- `sort_order`
- `created_at`
- `updated_at`

### 4.6 节假日 `holiday_calendar`

建议字段如下：

- `date`
- `day_type`
- `name`
- `source`
- `updated_at`

说明：

- `day_type` 至少支持：工作日、节假日、周末但调休上班。

### 4.7 同步元数据 `sync_meta`

建议字段如下：

- `key`
- `value`
- `updated_at`

用于保存：

- 最后同步时间
- 远端版本标识
- 本地数据摘要
- 同步配置项

## 6. 重复规则设计

### 6.1 支持的重复类型

- 按小时重复
- 按天重复
- 按周重复
- 按月重复
- 按年重复

每种重复类型支持：

- 间隔值 `interval`
- 可选截止时间 `until`
- 无限循环

### 6.2 重复规则存储方式

建议将重复规则存储为结构化 JSON，并同时保留关键字段用于查询优化。

例如：

- `recurrence_type`
- `recurrence_interval`
- `recurrence_rule_json`
- `recurrence_until`

理由如下：

- 基础查询条件可通过扁平字段快速过滤。
- 特定规则扩展可通过 JSON 承载。
- 避免后续需求调整时频繁改表。

### 6.3 实例展开策略

查询某个时间窗口内任务时，不从数据库读取“未来所有实例”，而采用以下过程：

1. 读取与查询窗口可能相关的模板版本段。
2. 依据重复规则，仅展开与查询窗口有交集的实例。
3. 为每个实例生成稳定的 `occurrence_key`。
4. 读取对应覆盖记录。
5. 将模板实例与覆盖合并，得到最终展示对象。

### 6.4 `occurrence_key` 设计

建议由以下信息构成：

- `series_id`
- `revision_id`
- 原始计划开始时间
- 原始计划截止时间

可编码为稳定字符串。

设计目标：

- 模板原始实例有唯一标识。
- 单次覆盖可以稳定绑定到某一实例。
- 模板变更时，历史实例和已存在覆盖不被误绑定。

## 7. 模板修改与版本段机制

### 7.1 模板修改流程

当用户选择“修改模板并影响未来”时：

1. 选定生效日期。
2. 查找当前生效中的版本段。
3. 将旧版本段的 `effective_until` 截断到生效日前一刻。
4. 新建一个版本段，从生效日期开始生效。
5. 历史实例保持按旧版本解释。

### 7.2 默认生效日期

- 默认取“下一次任务开始时间”。
- 用户可以手动调整。

### 7.3 未来已存在覆盖的处理

默认策略：

- 保留未来已存在的覆盖。

可选策略：

- 清除未来覆盖，使其重新按新模板解释。
- 将部分未来覆盖转为独立单次任务或独立模板。

详细交互在 UI 设计阶段补充，但领域层需要预留对应能力。

## 8. 单次覆盖机制

### 8.1 支持覆盖的内容

单次实例允许覆盖以下字段：

- 开始时间
- 截止时间
- 危险日
- 标题
- 备注
- 标签
- 优先级
- 状态

### 8.2 覆盖合并规则

合并顺序如下：

1. 按模板版本段生成原始实例。
2. 计算默认危险日。
3. 读取覆盖记录。
4. 使用覆盖字段替换实例对应字段。
5. 输出最终展示对象。

规则要求：

- 覆盖优先于模板。
- 覆盖只影响当前实例。
- 覆盖可递归编辑，但最终始终只落为一条该实例的覆盖记录。

### 8.3 跳过本次

“跳过本次”不单独建模，直接写入覆盖状态：

- `status = cancelled`

## 9. 危险日计算设计

### 9.1 基本规则

危险日以截止时间为锚点倒推。

支持单位：

- 小时
- 天

### 9.2 自然日倒推

示例：

- 截止时间为 2026-04-10 18:00
- 危险规则为提前 3 天
- 则危险日为 2026-04-07 18:00

### 9.3 工作日倒推

若启用工作日计算，则倒推过程中跳过：

- 法定节假日
- 普通周末

但保留：

- 调休工作日

### 9.4 单次实例危险日调整

若用户对某次实例手动修改危险日，则：

- 不再重新按模板规则计算该实例危险日
- 其他实例保持原模板规则

## 10. 查询与投影

### 10.1 日历视图查询

日历视图采用按窗口增量查询策略。

查询流程：

1. 计算当前可视窗口对应日期范围。
2. 向前后额外预取一定缓冲区间。
3. 展开该区间内所有相关实例。
4. 按天聚合任务展示数据。
5. 渲染为纵向日期流。

理由：

- 避免 1970 到 9999 全量预生成。
- 保证滚动性能可控。

### 10.2 近期视图查询

近期视图固定查询：

- 当前时间起未来 31 天

查询结果需包括：

- 单点任务
- 截止型任务
- 重复任务在该区间内的实例

### 10.3 近期视图排序

建议排序键如下：

1. 状态分组
2. 优先级
3. 危险日是否已到达及其临近程度
4. 截止时间
5. 开始时间
6. 创建时间

推荐排序规则：

- 未完成在前
- 完成和取消在后
- 高优先级在前
- 危险日更近的在前
- 截止时间更近的在前

### 11.1 页面结构

建议首版主要包含以下页面：

- 主日历页

### 11.2 日历视图渲染原则

- 按天渲染卡片或分组区域。

- 同日任务允许撑开高度。
- 单点任务使用点状或短条。

- 危险日当天在对应日期强调颜色。

### 11.3 编辑页设计原则

编辑任务时需支持以下操作：

- 新建单次任务
- 新建重复任务

- 修改模板并影响未来
- 选择模板生效日期
- 对未来覆盖记录选择保留或清除

## 12. 同步设计

### 12.1 同步模式

采用“远端优先覆盖本地”的简化策略。

### 12.2 同步触发时机

- 应用启动时
- 应用进入前台时
- 用户手动触发时
- 用户保存修改前

### 12.3 保存前检查流程

1. 用户发起保存。
2. 先执行远端检查。
3. 若远端未变动，则提交本地修改并上传。
4. 若远端已变动，则放弃本次修改，并以远端最新数据刷新本地。

### 12.4 数据组织建议

WebDAV 远端建议使用单一数据快照文件或少量固定文件，而不是拆分为大量小文件。

原因：

- 简化同步逻辑
- 减少文件级冲突
- 便于版本校验和整体覆盖

### 12.5 同步风险

该策略的代价是：

- 并发编辑时，本地最新修改可能被放弃

这是需求明确接受的行为，目的是换取实现简单和一致性高。

- 判断某日是否节假日

## 14. 后台与提醒设计

### 14.1 Windows

Windows 端需支持：

- 常驻后台
- 托盘入口
- 后台同步
- 本地提醒

### 14.2 Android

Android 端需支持：

- 应用前台使用
- 本地提醒
- 合理范围内的后台刷新

### 14.3 提醒与同步的关系

提醒依赖本地计算结果，不应强依赖实时联网。

建议流程：

1. 同步更新本地任务数据。
2. 本地重新计算近期危险日与截止任务。
3. 更新本地提醒计划。

## 15. 性能设计

### 15.1 计算策略

- 只展开查询窗口内的实例
- 对近期查询结果进行短期缓存
- 对节假日计算结果做缓存

### 15.2 渲染策略

- 日历视图使用虚拟化列表
- 只渲染可视区附近日期
- 同日任务多时允许撑开，但仍需控制单日组件构建成本

### 15.3 存储策略

- 使用 SQLite 持久化本地数据
- 不落库存储全部未来实例
- 仅对模板、版本段、覆盖、标签、节假日和同步元数据做持久化

## 16. 错误处理与边界场景

### 16.1 时间边界

- 全天任务与定时任务需统一转换为内部标准时间表示
- 需处理秒级精度
- 需避免跨时区造成截止时间错位

### 16.2 模板边界

- 生效日期前后的实例必须严格按对应版本段解释
- 历史实例禁止被未来模板修改污染

### 16.3 同步边界

- 本地保存时若远端发生变化，则本次保存失败
- 失败后需明确提示用户，说明本地视图已被远端刷新
